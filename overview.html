Provides implementation for general positioning, coordinate systems, and coordinate transformations. SEAGIS is an open source implementation of a subset of OpenGIS® Coordinate Transformation Services Specification. Being OpenGIS-compliant should allow developers to switch between this free implementation and any commercial implementation with no code change (providing that commercial implementations are OpenGIS compliant).

 

Packages organization

A SEAGIS's goal is to ensure interoperability with standard Java packages like Java2D and Java Advanced Imaging. However, it is not always possible to leverage Java classes and be OpenGIS-compliant in the same time. SEAGIS workaround the problem by providing two sets of packages:

  • The org.opengis package is the standard OpenGIS specification. Classes in this package are mostly interfaces, with no implementation. Interfaces defined there are well suited for RMI use, which implies that every method is declared to throws {@link java.rmi.RemoteException}. Developers looking for strict OpenGIS compliance and/or RMI use should look at this org.opengis package.
     
  • The net.seas.opengis package is an org.opengis mirror tuned for Java and local use. It contains concrete classes (not interfaces). There is generally a 1 to 1 correspondence between OpenGIS interfaces and SEAGIS classes, so developers familiar with one package should be familiar with the other. SEAGIS methods do not thrown RemoteException, which make them easier for local use but is not suitable for RMI use. Developers looking for Java2D interoperability and/or local use may look at this net.seas.opengis package.
     

Factory classes allow the creation of OpenGIS objects (RMI enabled) backed by SEAGIS classes. Internally, interoperability between OpenGIS interfaces and SEAGIS classes is assured through a set of RMI-aware adapters. Classes not related to any OpenGIS API appear in separated packages (e.g. net.seas.image.io).

 

Differences between SEAGIS and the OpenGIS specification

  • The OpenGIS specification requires that every class and interface name begin with a two letters prefix (e.g. {@link org.opengis.cs.CS_CoordinateSystem}, {@link org.opengis.ct.CT_CoordinateTransformation}, etc.). Those prefix are redundant with package name and are omitted in SEAS classes.

  • Made all enumeration classes immutable. Enumeration constants are instances of the actual enumeration class instead of integers. Names are uppercase, with no prefix like CT_TT, CT_DF or CS_AO.

  • Provide various methods and constructors to ease interoperability with the Java2D geometric package.

  • Changed signature of various transform(...) methods in {@link net.seas.opengis.ct.MathTransform}. New signatures are similar to {@link java.awt.geom.AffineTransform}'s transform methods, which should be familiar to many Java programmers. They offer better performance for local use (as opposed to RMI use). Note that the OpenGIS's signatures are better suited for RMI use.

  • SEAGIS provides localization support. Methods defined as getName() by OpenGIS are redifined as getName({@link java.util.Locale}) by SEAGIS.

  • {@link net.seas.opengis.cs.Ellipsoid#getInverseFlattening} returns {@link java.lang.Double#POSITIVE_INFINITY} instead of 0 for perfect spheres. Infinity is the mathematically correct value.

  • {@link net.seas.opengis.ct.CoordinateTransform} is set as a {@link net.seas.opengis.ct.MathTransform}'s subclass. Renamed CoordinateTransformation to CoordinateTransform for consistency.
