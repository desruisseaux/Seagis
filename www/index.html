Summary

One SEAGIS's goal is to implements a subset of OpenGIS® Coordinate Transformation Services Specification and OpenGIS® Grid Coverages Implementation Specification. OpenGIS is a consortium of organizations developing public specifications through structured committee programs. Products and services that conform to those open interface specifications enable users to freely exchange and apply spatial information, applications and services across networks, different platforms and products. A second SEAGIS's goal is to ensure interoperability with standard Java packages like Java2D and Java Advanced Imaging. SEAGIS extends existing Java classes like Format for parsing and formatting angles, and ImageReader for reading ASCII and binary files.

The current library supports Cylindrical Mercator, Transverse Mercator, Lambert Conformal Conical and Stereographic projections. Full source code is freely available under LGPL license, which is similar but somewhat less restrictive than GPL (see What is the LGPL?, How can GPL code be incorporated into the Communicator code base?).

Note: The "OpenGIS" part of this project moved to Geotools 2, which aim to support a wider range of OpenGIS specifications. The "SEAGIS" project hosted on this page is now a snapshot of the following Geotools's modules:

  • cts-coordtrans (Coordinate Transformation Services)
  • gcs-coverage (Grid Coverage Services)
  • renderer (a Java2D renderer)
  • gui (Graphics User Interface)
  • resources (Miscellaneous utilities)

 

 

Requirements

The SEAGIS download requires the following software:

  • J2SE 1.4 or later.
  • Java Advanced Imaging 1.1 or later.
  • The vecmath package from Java3D.
    In our understanding of JSR-148 (3D Media Utilities), the requirement for Java3D may be removed in a future version since it will retrofit into a requirement for JAI 1.2.

In addition, an optional experimental package can be downloaded separately. This experimental package contains modules that are not part of OpenGIS's specification, but related to geographic information. The experimental package has higher requirement than the SEAGIS package, as noted below. Note that you do not need those requirements if you download only the SEAGIS package.

  • J2SE 1.4
  • Prototype compiler for JSR014: Adding Generics to the Java Programming Language

 

 

Browse contents and download

This project has two main modules: an opengis implementation (SEAGIS) and an experimental one. The SEAGIS module contains a copy of OpenGIS interfaces. Those interfaces are slightly different from the ones currently available on OpenGIS web site, in that most methods are declared to throw RemoteException. This detail seems to have been forgotten in the original OpenGIS's interfaces, but we don't know if and when it will be fixed.

Note that Seagis's development has moved to Geotools 2. The current Seagis download is just a snapshot of some Geotools 2 modules.

Changes

Current version is alpha-0.6.3. Changes since last version include the following:

  • Bug fixes (Note: the Coordinate Transformations Services are know to not pass fully OpenGIS tests. We are still working on it).
  • Some packages from the experimental project has moved to Geotools, namely org.geotools.axis and org.geotools.renderer.
  • New GridCoverage operations ("Recolor", more complete "Resample").

Changes between alpha-0.6.2 and alpha-0.6.0 include:

  • Well Know Text (WKT) parsing has been implemented
    • .
  • A MathTransform backed by a localization grid has been added. Localization grid are used in remote sensing for transforming coordinate from a system tied to the spacecraft into geographic coordinates.
  • The EPSG factory now return coordinate system with fractional degrees even if it was declared as sexagesimal degrees in the EPSG database. Sexagesimal degrees are really unconvenient in any computation process. User still able to use sexagesimal degrees if he really want it.

Changes between alpha-0.6.0 and alpha-0.5.2 include:

  • Orthodromic distance computation has been improved. The old implementation used an approximation for spherical earth. New implementation use an ellipsoidal model. This a port from Fortran to Java of the code found at ftp://ftp.ngs.noaa.gov/pub/pcsoft/for_inv.3d/source/inverse.for.
  • Special optimizations has been added for MathTransform in 1D case, since the new API in GCS depends heavily on it.
  • A "Logarithmic" transform has been added. This is needed by GCS, since SeaWiFS images (ocean color) from eosdata.gsfc.nasa.gov use a logarithmic transform.
  • A first draft of EPSG factory (for Coordinate Systems).
  • The helper class CategoryList is no longer public. Its functionalities has been moved into SampleDimension.
  • According OpenGIS specification, the SampleDimension class has two attributes ("scale" and "offset") which allow transforming sample values (usually integers) into geophysics values (for example height in meters) using a linear relation (height = offset + scale*sample). This is not enough in some cases; for example SeaWiFS images use a logarithmic transformation. In the old GCS API, using a non-linear relation required subclassing Category class and overriding toSample and toGeophysics methods. In the new API, methods toSample and toGeophysics don't exist anymore. The user can construct categories with arbitrary MathTransform1D objects instead.
  • In the old API, some methods expected a boolean argument (e.g. Category.getRange(boolean), GridCoverage.getRenderedImage(boolean)). In the new API, the boolean argument don't exists anymore. A single method, geophysics(boolean), returns a "geophysics" view of an object instead. This solution simplify the API (remove the boolean argument from many methods) and is more general.
  • Methods like toGeophysics(RenderedImage, ...) has been removed. Instead, a "GC_SampleTranscoding" operation has been registered through the normal JAI operations registry. This operation is used internally for transcoding an image from sample values to geophysics values (e.g. height in meters) and conversely.
