Coordinate Transformations implementation. Transformations (for example map projections) are performed by {@link MathTransform}. Instances of {@link MathTransform} are created by {@link MathTransformFactory}. For example, a mercator cylindrical projection can be created by the following code:

{@link net.seas.opengis.ct.MathTransform} projection = {@link net.seas.opengis.ct.MathTransformFactory#DEFAULT}.createParameterizedTransform("Mercator_1SP", null, null);

The "Mercator_1SP" string is the classification name for the mercator cylindrical projection. This package support the following classification names:

Differences with the OpenGIS specification

  • Omitted the CT_ prefix for all classes and interfaces.
  • Made all enumeration classes immutable. Enumeration constants are instances of the actual enumeration class instead of integers. Names are uppercase, with no CT_TT or CT_DF prefix.
  • Classes CoordinateTransformationFactory and CoordinateTransformationAuthorityFactory are currently not implemented.
  • The following methods are curently not implemented. They may be implemented in a future version:
    • MathTransform.derivative(CoordinatePoint)
    • MathTransform.getDomainFlags(ConvexHull)
    • MathTransform.getCodomainConvexHull(ConvexHull)
    • MathTransform.getWKT()
    • MathTransform.getXML()
    • MathTransformFactory.createAffineTransform(...)
    • MathTransformFactory.createPassThroughTransform(...)
    • MathTransformFactory.createFromWKT(...)
    • MathTransformFactory.isParameterAngular(...)
    • MathTransformFactory.isParameterLinear(...)
  • Changed signature of various transform(...) methods in {@link net.seas.opengis.ct.MathTransform}. New signatures are similar to {@link java.awt.geom.AffineTransform}'s transform methods, which should be familiar to many Java programmers. They offer better performance for local use (as opposed to RMI use). Note that the "old" signatures in the standard OpenGIS interfaces are better suited for RMI use than the "new" ones.
