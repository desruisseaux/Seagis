Differences between SEAGIS's API OpenGIS specification

The net.seas.opengis package and sub-packages are basically a org.opengis's mirror. Most classes have a 1 to 1 equivalent. However, the SEAGIS's API is modified in order to make it more "Java friendly" and ease interoperability with other Java packages (specially Java2D and Java Advanced Imaging). The main differences are listed below:

Naming convention

The OpenGIS specification requires that every class and interface name begin with a two letters prefix (e.g. CS_ in org.opengis.cs.CS_CoordinateSystem and CT_ in org.opengis.ct.CT_CoordinateTransformation). Those prefix are redundant with package name and are omitted in SEAS classes.

Enumeration classes

The OpenGIS's specification define enumeration classes, but use them only for parameters and return values in method calls. Enumeration constants are declared as integer. In the SEAGIS's API, all enumeration classes are immutable. Ennumeration constants are instances of the actual enumeration class instead of integers. Names are uppercase, with no redundant prefix like CT_TT, CT_DF or CS_AO.

Localization support

SEAGIS provides localization support. Methods defined as getName() by OpenGIS are redefined as getName(Locale) by SEAGIS.

Classes hiearchy changes

  • The CoordinateTransform class has been moved as a MathTransform's subclass (since coordinate transforms are kind of math transforms with additional semantic). MathTransform is itself a Info's subclass (since it shares many common methods). Note that CoordinateTransformation has been renamed CoordinateTransform for consistency with this new hierarchy.

Method's name changes

  • OpenGIS's getMetadataValue(String) methods are functionally equivalent to Java's getProperty(String) methods. SEAGIS follow the Java's convention (i.e. "MetadataValue" has been renamed "Property") in order to ease interoperability with interfaces like javax.media.jai.PropertySource.

Method's signature changes

  • Changed signature of various MathTransform.transform(...) methods. New signatures are similar to AffineTransform's transform methods, which should be familiar to many Java programmers. They offer better performance for local use (as opposed to RMI use). Note that the OpenGIS's signatures are better suited for RMI use.

Method's behavior changes

  • Ellipsoid.getInverseFlattening() returns Double.POSITIVE_INFINITY instead of 0 for perfect spheres. Infinity is the mathematically correct value.
